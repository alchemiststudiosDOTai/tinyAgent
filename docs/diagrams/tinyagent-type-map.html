Key takeaway

Pydantic models carry the actual conversation payloads. The unions like Message, AgentMessage, and AssistantContent are the connective tissue that makes those payloads composable without dropping type specificity.

Foundation notes
  • JsonValue is recursive and underpins tool arguments, usage, and details.
  • dump_model_dumpable() enforces the model_dump() contract at provider boundaries.
  • ThinkingLevel and ThinkingBudgets control reasoning behavior separately from message payloads.
Section 2

Content + message hierarchy

This is the heart of the module. The content models are small, focused building blocks; the message models define which blocks are legal in which role.

Content blocks
TextContent
  • type="text"
  • text payload
  • optional signature + cache control
ImageContent
  • type="image"
  • url and mime_type
  • used on user and tool-result sides
ThinkingContent
  • type="thinking"
  • reasoning text + signature
  • assistant-only content
ToolCallContent
  • id + name
  • arguments: JsonObject
  • partial_json for streaming
Role models
UserMessage

Only TextContent and ImageContent; no thinking blocks, no tool calls.

AssistantMessage

Carries AssistantContent | None plus stop_reason, provider metadata, usage, and error_message.

ToolResultMessage

Turns AgentToolResult back into conversational state via text/image content and a JSON details payload.

CustomAgentMessage

Escape hatch for internal roles. It lives in AgentMessage but not in the provider-facing Message union.

Union aliases that define the hierarchy
high signal lines
ToolCall = ToolCallContent
AssistantContent = TextContent | ThinkingContent | ToolCallContent
Message = Union[UserMessage, AssistantMessage, ToolResultMessage]
AgentMessage = Union[Message, CustomAgentMessage]
Boundary rule: Message means “safe to hand to the LLM layer.” AgentMessage means “safe inside the agent runtime,” which is why it can additionally carry custom internal roles before conversion.
Section 3

Tools + streams

Outside the message hierarchy, the file defines the contracts that make the loop executable: tool definitions, provider invocation inputs, streaming outputs, and agent loop state.

Tool types
  • Tool = static definition: name, description, parameters
  • AgentTool = Tool plus label and async execute
  • AgentToolResult = text/image content plus JSON details
  • AgentToolUpdateCallback streams partial tool output
Context and model
  • Context carries provider-safe Message values
  • AgentContext carries broader AgentMessage values
  • Model selects provider, id, api, thinking level
  • SimpleStreamOptions adds api key, temperature, max tokens, signal
Loop state
  • AgentLoopConfig wires callbacks and model-level execution settings
  • AgentState stores system prompt, tools, messages, pending tool calls, error
  • EventStream is the async queue that yields AgentEvent and returns final messages

Lifecycle grouping
  • Agent: one run from start to end
  • Turn: one assistant response plus tool follow-through
  • Message: streaming phases for a single assistant message
  • Tool execution: start, partial updates, completion

Section 5

Mental model

If you only remember one map, remember this one: Pydantic models are the data, dataclasses are the lifecycle, and aliases/protocols are the glue that keep the runtime composable.

Read the module in layers
Layer 1

Foundations. JSON aliases, model-dump contract, thinking configuration.

Layer 2

Payloads. Content, messages, tools, contexts, models, stream events, state.

Layer 3

Lifecycle. Agent events, loop config, queue sentinels, event stream orchestration.

Public surface exposed by tinyagent
Messages UserMessage, AssistantMessage, ToolResultMessage Content TextContent, ImageContent, ToolCallContent Events AgentEvent and the 10 concrete event dataclasses Loop AgentLoopConfig, AgentState, EventStream Tools Tool, AgentTool, AgentToolResult Providers StreamFn, StreamResponse, proxy stream types

Practical consequence: most users can stay inside the re-exported surface from tinyagent and only open agent_types.py when they need deeper reasoning about runtime shape.

Compressed summary: TextContent, ImageContent, ThinkingContent, and ToolCallContent are the atoms. UserMessage, AssistantMessage, and ToolResultMessage are the first meaningful molecules. Context and Model feed providers. AssistantMessageEvent streams the reply. AgentEvent narrates the whole run around that core exchange.
