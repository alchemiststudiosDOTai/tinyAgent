How the Rust streaming bridge actually works

TinyAgent’s Rust-backed provider path is a two-language bridge: Python shapes the call, Rust owns the long-lived stream and SSE parsing, and Python rehydrates the stream back into canonical AssistantMessageEvent and AssistantMessage models.

Core insight

Rust owns the live stream, Python owns the public contract. The handle returned by openai_completions_stream() is intentionally blocking on the Rust side; the async illusion is created in Python by calling next_event() and result() inside asyncio.to_thread().

What this buys you
  • SSE parsing and event serialization happen off the Python hot path.
  • The agent loop still receives the same canonical TinyAgent event/message models.
  • Provider differences collapse into one StreamResponse-shaped adapter.
Section 2

Call path

The streaming call is not “Python calls Rust and gets async back.” It is “Python asks Rust to start work, Rust emits blocking handle methods, and Python wraps those methods in async threads.”

Python wrapper side
  • stream_alchemy_openai_completions() resolves provider, base URL, API, and API key.
  • It serializes Model, Context, and SimpleStreamOptions into plain dicts.
  • It returns an AlchemyStreamResponse, which satisfies TinyAgent’s StreamResponse protocol.
Rust handle side
  • OpenAICompletionsStream.next_event() blocks on a standard mpsc::Receiver.
  • OpenAICompletionsStream.result() reads the final message from a separate receiver once.
  • Both methods use py.allow_threads so Python is not holding the GIL while Rust waits.
Why two channels? Events and the final assistant message have slightly different lifecycles. The event stream may end at the same moment a terminal message is captured, but callers still need a clean, dedicated path for result() even after iteration finishes.

Python-side call shape
tinyagent/alchemy_provider.py
model_dict = {
    "id": model.id,
    "provider": provider,
    "api": api,
    "base_url": base_url,
    "name": name,
    "headers": headers,
}
context_dict = {
    "system_prompt": context.system_prompt or "",
    "messages": [dump_model_dumpable(m, where="context.messages") for m in context.messages],
    "tools": _convert_tools(context.tools),
}
handle = alchemy_llm_py.openai_completions_stream(model_dict, context_dict, options_dict)
Routing rules
  • Explicit model.api wins after alias normalization.
  • Blank api is inferred from provider.
  • minimax and minimax-cn map to minimax-completions.
  • Everything else defaults to openai-completions.
Auth rules
  • options.api_key has highest priority.
  • Otherwise Python resolves env-key fallback from provider.
  • Rust receives the already selected API key in options_dict.
Section 4

Live stream bridge

The most important function in the Rust file is not the request builder — it is the background task that pulls stream.next().await, captures the terminal message, and fans results back to Python.

Tokio runtime
  • A single lazily initialized multi-thread runtime backs all streaming work.
  • openai_completions_stream() calls RUNTIME.enter() before provider code that may spawn tasks.
  • The actual event loop is then launched with RUNTIME.spawn(async move { ... }).
Event fan-out
  • Each provider event is converted by event_to_py_value().
  • Converted JSON values go through tx.send(...) to the event receiver.
  • The terminal AssistantMessage is separately cached and sent on final_tx.
Python async façade
  • AlchemyStreamResponse.__anext__() awaits asyncio.to_thread(self._handle.next_event).
  • AlchemyStreamResponse.result() does the same with the final result method.
  • Returned dicts are validated back into AssistantMessageEvent and AssistantMessage.
Rust-side streaming loop
src/lib.rs · condensed
RUNTIME.spawn(async move {
    let mut terminal = None;

    while let Some(ev) = stream.next().await {
        match &ev {
            Done { message, .. } => terminal = Some(message.clone()),
            Error { error, .. } => terminal = Some(error.clone()),
            _ => {}
        }

        if tx.send(event_to_py_value(&ev)).is_err() {
            return; // Python side dropped the stream
        }

        if matches!(ev, Done { .. } | Error { .. }) {
            break;
        }
    }

    let _ = final_tx.send(final_message_json);
});
Subtle but important: every non-terminal event still carries a partial assistant message snapshot. That means Python consumers do not need to reconstruct the whole message from deltas unless they want to — the provider already ships the rolling partial state alongside the delta.

Events, terminals, and safety contracts

The stream is only useful because it is normalized. Both the event vocabulary and the final message contract are enforced before the rest of TinyAgent sees them.

Terminal events are done and error. Rust maps provider-specific stop reasons into TinyAgent’s canonical names before Python validation runs.

Terminal mapping
  • Done::Stop → reason="stop"
  • Done::Length → reason="length"
  • Done::ToolUse → reason="tool_calls"
  • Error::Error → reason="error"
  • Error::Aborted → reason="aborted"
Surface Kind What it guarantees
event_to_py_value() rust serializer Normalizes every provider event into a TinyAgent-shaped JSON payload, including rolling partial state.
AlchemyStreamResponse.__anext__() python adapter Turns a blocking Rust handle method into async iteration and validates the result as AssistantMessageEvent.
_validate_assistant_message_contract() python validation Ensures the final result is a valid AssistantMessage; optionally requires full usage data.
_validate_usage_contract() usage schema Checks canonical keys like input, output, total_tokens, and nested cost fields.
Two public Rust entrypoints
  • openai_completions_stream() returns a stream handle for incremental reads.
  • collect_openai_completions() consumes the full stream and returns {events, final_message}.
  • Both route through the same request-building and alchemy dispatch path.
Current constraints
  • Only openai-completions and minimax-completions are dispatched.
  • Image blocks are not supported by this Rust path yet.
  • The bridge is real-time but not natively async end-to-end; Python uses worker threads around blocking handle methods.
Mental model: alchemy_provider.py is the adapter, src/lib.rs is the stream engine, alchemy_llm::stream() is the transport abstraction, and AlchemyStreamResponse is the compatibility layer that makes the whole Rust path look like any other TinyAgent StreamFn implementation.
