# T002: Rebuild the `_alchemy` import and stream-handle contract

**Summary**: Make the Rust binding expose the same load and stream-handle behavior the Python adapters depend on, including module import fallback, `openai_completions_stream(...)`, blocking `next_event()`, and blocking `result()`.

**Owner**: runtime

**Estimate**: 6h

**Dependencies**: T001

**Target milestone**: M2

**Acceptance test**: `uv run pytest tests/test_alchemy_provider.py tests/test_rust_binding_provider.py -q`

**Files/modules touched**:
- rust/src/lib.rs
- rust/src/types.rs
- tinyagent/alchemy_provider.py
- tinyagent/rust_binding_provider.py
- tests/test_alchemy_provider.py
- tests/test_rust_binding_provider.py

**Steps**:
1. Implement the binding entrypoint and returned handle shape so both Python adapters can call `openai_completions_stream(model, context, options)` without adapter-specific shims.
2. Preserve the current import fallback behavior for `tinyagent._alchemy` versus `_alchemy` in each provider module instead of collapsing them into a new load order.
3. Keep Python-side iteration and final-result retrieval compatible with `asyncio.to_thread(...)` around blocking native calls.
