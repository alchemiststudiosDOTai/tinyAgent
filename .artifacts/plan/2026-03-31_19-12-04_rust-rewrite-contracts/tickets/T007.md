## Risks & Mitigations

- Provider drift risk: keep both adapter modules under test while the Rust binding changes, rather than replacing one adapter first and hoping the other still works.
- Event-shape drift risk: treat `tests/test_contracts.py` and `docs/harness/tool_call_types_harness.py` as parity gates for streamed event names and terminal payloads.
- Hidden release regressions: keep release scripts and wheel tests in the same milestone as packaging changes instead of delaying them until after runtime work.
- Doc/runtime mismatch: update `docs/api/` and `docs/releasing-alchemy-binding.md` in the same tasks that change the corresponding contract.

## Test Strategy

- Use the provider tests to pin import/load order, payload serialization, and final-message validation.
- Use usage and contract tests to pin stop reasons, assistant event shapes, and required `usage` keys.
- Use agent and tool execution tests to pin event ordering and message/result behavior.
- Use the harness and release tests as final proof that the rewrite still satisfies the repo's operator-facing contract.
